Chapter 02

1. 第一课：先认识 Codex

Codex 可以理解成一个会读资料、改文件、跑命令、看网页、检查结果的工作助手。

它最适合处理这类任务：

  • 有明确目标。
  • 有资料或文件。
  • 有步骤。
  • 能检查结果。
  • 需要反复修改。

它不适合处理这类任务：

  • 只说“帮我看看”。
  • 不给资料，却希望它知道公司内部情况。
  • 没有验收标准，只希望它猜中你想要什么。
  • 结果不能检查，只能凭感觉说好不好。

1.1 Codex 和普通聊天有什么不同

普通聊天Codex
更像问答更像做事
你问一句，它答一句它可以读文件、改文件、跑检查
结果通常是一段文字结果可以是网页、文档、脚本、截图、部署链接
容易停在建议层面可以推进到可验收的交付物

1.2 员工只需要先记 5 个词

词傻瓜式理解例子
目标你到底要它做成什么把这份乱文档改成员工教程
材料它需要读什么Notion、飞书、截图、图片、旧文档
边界哪些不要做不要搬无关案例，不要改掉我原来的表达
过程它准备怎么做先读资料，再拆结构，再写正文，再检查页面
验收怎么判断做对了导航能跳、图片正常、手机不挤、文字易懂

1.3 第一次练习

选一个小任务，不要太大。

适合第一次练习：

  • 把一页会议记录整理成 SOP。
  • 把一份旧教程改成新人版。
  • 把一个网页检查出 5 个排版问题。
  • 把一个文件夹里的资料整理成目录。
  • 把一段操作视频整理成步骤。

不适合第一次练习：

  • 一上来做完整系统。
  • 一上来接复杂外部工具。
  • 一上来让它全自动发布到线上。
  • 一上来让它改很多未知文件。

1.4 本课交付物

写一张“我可以交给 Codex 的任务清单”。

岗位任务材料在哪里想要的结果怎么验收
新人培训文档重写旧文档、截图、反馈新人能照着做的教程新人不问人也能完成第一遍
网页最终检查本地网页、参考图问题清单和修复版电脑端和手机端都正常
资料整理飞书、Notion、图片结构化学习页目录清楚，图片能解释步骤
Chapter 03

2. 第二课：第一次上手

第一次上手不要追求高级。

目标是跑通一个完整闭环：

  1. 交代任务。
  2. 给它材料。
  3. 让它先复述计划。
  4. 让它执行。
  5. 看结果。
  6. 指出问题。
  7. 让它修改到能用。

2.1 第一步：先说清你要什么

不要这样说：

帮我优化一下。

要这样说：

把这份员工笔记改成 Codex 上手教程。读者是完全没用过 Codex 的员工。要求短句、步骤清楚、图片放在能帮助理解的位置。不要改掉我专门写的表达。最后检查电脑端和手机端。

这不是让大家复制这句话。

重点是学会把“目标、读者、材料、边界、验收”说出来。

2.2 第二步：把材料给够

Codex 不是公司内部资料库。

你不给材料，它只能猜。

常见材料有：

材料类型怎么给注意点
文档Notion、飞书、Markdown、Word、PDF告诉它哪些内容保留，哪些删掉
截图页面截图、报错截图、参考图截图要清晰，最好说明你关心哪里
网页本地地址或线上地址让它实际打开检查，不要只口头判断
文件夹项目目录、图片目录、导出目录让它先列结构，再动手
旧版本旧网页、旧文案、旧教程告诉它不能丢掉哪些原意

2.3 第三步：让它先复述计划

第一次不要直接让它大改。

先让 Codex 复述：

  • 它理解的目标是什么。
  • 它准备保留哪些内容。
  • 它准备删除哪些内容。
  • 它准备怎么分章节。
  • 它准备怎么验收。

如果复述不对，先纠正。

不要等它写完一大堆再返工。

2.4 第四步：执行时看证据

你要看它做了什么，而不是只看它说“完成了”。

能验收的证据包括：

任务证据
改网页本地预览截图、手机截图、文件路径
改文档修改后的文档、前后对比、章节目录
跑测试命令输出、通过/失败原因
整理图片图片路径、缺失资源检查
部署网站线上链接、commit、资源 200 检查

2.5 第五步：继续追问，不要客气

你可以狠狠 push，但要 push 具体问题。

有效追问：

  • 这段还是像说明书，改成新人能照着做。
  • 这个标题太长，换成更短的教程标题。
  • 这个案例和课程关系不清楚，补一句“这个案例学什么”。
  • 手机端第二行只有两个字，重新排一下。
  • 你不要重新编内容，回到原文和图片。

无效追问：

  • 再好一点。
  • 高级一点。
  • 像别人那样。
  • 你自己看着办。

2.6 本课交付物

完成一次小任务，并留下记录：

记录项写什么
任务目标我要 Codex 做什么
输入材料我给了哪些文件、链接、截图
执行过程它先做了什么，后来改了什么
验收证据截图、测试、链接、文件路径
下一次改进下次交代任务要补什么
Chapter 04

3. 第三课：把任务交代清楚

这一课不是教大家背提示词。

真正要学的是：怎么把工作交代成 Codex 能执行的任务。

3.1 一次任务只说 6 件事

要素怎么写例子
目标最终要交付什么把飞书资料改成 Codex 员工教程网站
读者给谁看给完全没用过 Codex 的员工看
材料依据是什么原 Notion、飞书截图、参考图片、现有网页
边界不要做什么不要搬无关内容，不要只堆案例，不要改掉专门表达
风格读起来像什么傻瓜式、步骤清楚、短句、少 AI 味
验收怎么判断完成桌面手机正常、导航可跳、图片正常、案例完整

3.2 不要写成愿望，要写成动作

愿望式说法动作式说法
帮我优化一下先读全文，列出读者会卡住的地方，再按章节重写
做得好看一点检查桌面和手机布局，修掉右侧空白、图片压文字、标题别扭换行
内容全面一点按“认识、上手、工作流、工具、案例、团队落地”补齐
不要像说明书每节改成：先解释用途，再给步骤，再给检查点

3.3 让 Codex 先给你计划

适合用在大任务前：

先不要改文件。你先读现有网页和材料，告诉我现在像不像教程，哪里像提示词合集，哪里案例太散。然后给出你准备怎么改。

等方向对了，再让它执行。

3.4 什么时候可以让它直接干

这些任务可以直接干：

  • 修错别字。
  • 删掉指定词。
  • 把某一节改短。
  • 检查图片路径。
  • 重新生成网站。
  • 跑一次发布脚本。

这些任务先让它计划：

  • 重做整篇教程结构。
  • 把飞书一整套内容搬到网站。
  • 大改 UI。
  • 接 MCP 或外部系统。
  • 部署给员工使用。

3.5 本课交付物

写一张任务说明卡：

项目填写
我要做什么
读者是谁
材料在哪里
必须保留什么
必须删除什么
风格要求
验收方式
Chapter 05

4. 第四课：让 Codex 读懂你的资料

很多结果不好，不是 Codex 不会做。

是它没有拿到正确材料。

4.1 资料要按用途给

你给的东西Codex 能做什么你要补充什么
原文重排结构、改写、删减哪些话不能改，哪些地方读着累
图片放进教程、解释步骤图片用途，不要只说“放进去”
参考网站学习结构和节奏哪些要学，哪些不要照搬
项目文件改代码、生成页面、检查资源启动方式、发布方式、不能动的文件
云文档链接提取文字和图片只取当前页还是也看子页

4.2 先让它读，再让它改

大任务建议分两步：

  1. 先读材料，列结构和问题。
  2. 再改内容和页面。

这样能避免它没看完就开始编。

4.3 图片不是装饰，是降低理解成本

图片应该放在这些位置：

  • 第一次讲一个工具入口时。
  • 步骤容易说不清时。
  • 读者需要对照界面操作时。
  • 案例需要证明来源时。
  • 最终效果需要让人先看到时。

图片不应该这样用：

  • 为了好看随便放。
  • 和正文没有关系。
  • 一张图占很大篇幅但不解释。
  • 图片太糊，还让员工照着做。

4.4 本课交付物

做一张资料清单：

材料用途是否必须备注
原文保留内容和口吻必须不能重新编
参考截图学习结构和案例名称必须不要遗漏侧栏案例
贴纸调整风格和亲和力可选不要抢正文
飞书图片解释步骤必须有用才放
线上链接最终验证必须发布后检查
Chapter 06

5. 第五课：检查和验收

Codex 最重要的习惯：不要只听它说完成。

要看证据。

5.1 内容怎么验收

检查项通过标准不通过怎么改
是不是教程读者知道先做什么、再做什么补步骤和交付物
是不是太像说明书短句、动作明确、不绕拆段，删空话
案例有没有抢主线案例服务于学习目标把案例移到对应课程后面
有没有遗漏目录里的案例正文都有补齐缺的案例
有没有乱编基于原文、图片、截图回到原材料重写

5.2 网页怎么验收

检查项怎么测
电脑端看 1440 宽度有没有右侧大片空白、图片是否压文字
手机端看标题是否别扭换行、表格是否挤出屏幕
导航左侧目录能不能跳到对应章节
图片所有图片是否 200，是否清晰
线上版GitHub Pages 是否刷新，新增资源是否能打开

5.3 最容易漏的检查

  • 只检查本地，忘了检查线上。
  • 只看电脑端，忘了看手机。
  • 只看第一屏，忘了看中间案例区。
  • 只看文字，忘了图片路径。
  • 只看案例数量，忘了案例是不是能教人。

5.4 让 Codex 自己做 review

你可以让它站在读者角度检查：

你现在是第一次学 Codex 的员工。读完这页，你哪里会卡住？哪里像说明书？哪里像案例堆？哪些地方还需要步骤、图片或验收标准？不要只评价，要直接修改。

重点不是这句话本身。

重点是让 Codex 换成读者视角，而不是作者视角。

交一张验收表：

项目结论证据
桌面端通过 / 不通过截图路径
手机端通过 / 不通过截图路径
案例完整通过 / 不通过案例列表
图片资源通过 / 不通过检查命令输出
线上版本通过 / 不通过线上链接

6. 第六课：MCP 和工具

MCP 可以理解成：让 Codex 连接外部工具或系统的一种方式。

但不是所有任务都需要 MCP。

6.1 什么时候不需要 MCP

这些场景先不要接 MCP：

  • 只是改文案。
  • 只是整理图片。
  • 只是改网页布局。
  • 只是读取你已经给出的文件。
  • 只是生成一份静态教程。

先用普通文件、截图、网页检查就够了。

6.2 什么时候需要 MCP

这些场景才考虑 MCP：

  • 需要读取实时数据。
  • 需要写入外部系统。
  • 需要访问私有知识库。
  • 需要调用公司内部接口。
  • 需要把一个工具长期接给团队使用。

6.3 MCP 判断表

问题如果答案是“是”下一步
是否必须读实时数据可能需要工具先做只读查询验证
是否要写入外部系统风险更高先做 dry-run 和回滚方案
是否有权限和密钥必须明确不要把密钥写进教程
是否多人共用需要稳定部署考虑远程或代理架构
是否只是一次性任务不一定需要 MCP先用手动文件处理

6.4 Flomo MCP 案例学什么

Flomo MCP 的价值不是“看起来技术厉害”。

它适合教三个点：

  • 先验证 API，不要一上来写 MCP。
  • 先做最小写入，不要直接批量写。
  • 写入类工具必须有检查和回滚意识。

6.5 MCP 架构怎么选

先记一个顺序：

  1. 能不用 MCP，就先不用。
  2. 能本地验证，就先本地验证。
  3. 需要多人共用，再考虑远程。
  4. 需要权限和审计，再加代理。
  5. 需要长期维护，再写发布和运维文档。

6.6 本课交付物

交一张 MCP 判断表：

我要接的工具为什么需要读还是写第一步验证风险
知识库需要读最新资料读先查一份测试资料资料过期、权限
笔记工具需要写入笔记写先写一条测试笔记误写、重复写
查询工具需要实时查询读查一次测试数据接口失败、参数不全

7. 第七课：Skill：把经验沉淀下来

Skill 适合解决这类问题：

这件事以后还会反复做，而且有固定步骤、固定判断、固定检查。

7.1 Skill 不是什么

Skill 不是把一句提示词保存起来。

Skill 更像一份“可复用操作说明”。

它应该告诉 Codex：

  • 什么时候用这个 Skill。
  • 输入材料需要什么。
  • 按什么步骤处理。
  • 哪些内容不能乱改。
  • 用什么脚本或工具保证结果稳定。
  • 做完怎么验收。

7.2 什么时候该做 Skill

场景要不要做 Skill原因
只做一次的临时任务不要写清任务就行
每周都要整理同类文档要流程固定，适合复用
每次生成同一种 PPT要可以用模板和脚本稳定输出
还没跑通的流程先不要先手动跑通再沉淀
需要固定检查清单要能减少遗漏

7.3 录制视频整理 Skill

这是你前面专门强调的重点。

不要把视频只总结成一段话。

要把视频变成下一次还能复用的流程。

步骤做什么产出
1. 转写把视频转成文字，保留时间顺序原始文字稿
2. 分段按操作目标分成几段章节列表
3. 提动作每段提取点击、输入、判断步骤清单
4. 补判断把为什么这样做写出来判断规则
5. 补素材截关键画面或保留原图步骤图片
6. 写验收定义做完怎么看对不对检查清单
7. 试跑让 Codex 按 Skill 做一次可复用版本

7.4 Skill 草稿长什么样

字段要写什么例子
Skill 名称这套流程叫什么录制视频整理成教学 SOP
触发场景什么时候用给一段操作视频，让 Codex 生成教程
输入材料需要什么视频、字幕、关键截图、目标读者
处理步骤固定流程转写、分段、提动作、补判断、配图、验收
保留规则不能丢什么原操作顺序、工具名称、关键口径
删除规则不要什么闲聊、重复话、无关背景
验收标准怎么判断成功新人照着能完成一次

7.5 PPT Skill 案例学什么

PPT Skill 不是让模型随便排版。

它要解决的是“稳定生成文件”。

正确分工：

谁负责负责什么
人给主题、受众、页数、风格、素材
Codex拆大纲、组织内容、调用 Skill
脚本生成真实 PPTX 文件，保证版式稳定
验收打开文件检查字体、图片、页边距、目录

7.6 本课交付物

写一份 1 页 Skill 草稿。

要求：

  • 能看出触发场景。
  • 能看出输入材料。
  • 能看出步骤。
  • 能看出检查点。
  • 能看出哪些内容不能乱改。
  • 能让另一个人照着试跑。

8. 第八课：网页和浏览器验证

很多教程、网站、系统页面，不能只靠文字检查。

要让 Codex 打开页面，看真实效果。

8.1 浏览器验证适合做什么

场景Codex 要做什么证据
网页排版打开页面，看电脑端和手机端截图
按钮流程点击按钮，确认能进入下一步操作记录
表单输入测试数据，看提示是否正确截图和错误信息
导航点左侧目录，看是否跳转跳转结果
图片资源检查图片是否加载资源状态

8.2 不要只看第一屏

网页最容易漏的是中间部分。

比如：

  • 案例区表格在手机端横向挤出去。
  • 图片很大，但下面解释不清楚。
  • 标题换行后第二行只有一两个字。
  • 左侧导航能显示，但点不到对应章节。
  • 本地能看，线上图片 404。

8.3 Playwright 案例学什么

Playwright 适合做自动化网页检查。

它的价值不是炫技，而是让检查可重复。

检查动作怎么做通过标准
打开页面进入本地或线上 URL返回 200，页面非空
截图桌面和手机各截一张布局不乱
点击导航点击某个目录项滚动到对应章节
查资源检查图片路径没有缺图
查溢出检查是否横向撑开手机端不出现异常横向滚动

8.4 Chrome 案例学什么

Chrome 案例适合真实网页操作。

让 Codex 打开你的 Chrome，可以利用你的登录态、网页状态和真实浏览环境。

适合：

  • 检查需要登录的网站。
  • 看真实页面显示。
  • 复现你在浏览器里看到的问题。
  • 截图给你确认。

不适合：

  • 让它随便点未知按钮。
  • 让它处理没有授权的数据。
  • 让它在没有确认的情况下提交表单或发布内容。

检查项结果证据

手机端通过 / 不通过截图路径
导航跳转通过 / 不通过点击记录

9.1 案例一：Codex × 知识库问答：让回答有依据

教学案例案例一：Codex × 知识库问答：让回答有依据

知识库问答案例用来训练员工把“资料”变成“可回答的问题”。重点不是让 Codex 说得像人，而是让它回答有来源、有边界、能复查。

学什么

资料整理、知识库结构、引用来源、回答验收。适合客服 FAQ、产品手册、培训资料、内部制度。

准备什么

准备 3-10 份真实资料，最好包含产品说明、常见问题、旧回复、更新记录。先删掉明显过期和无关内容。

照着做

第一步，把资料按主题分组；第二步，让 Codex 提取每组的用途和关键词；第三步，整理成知识库页面；第四步，准备 10 个真实问题测试；第五步，检查回答有没有引用资料。

验收

回答能说出依据来自哪份资料；不知道时会说明“不确定”；不会编不存在的政策；同一个问题换个问法也能答到同一类资料。

交付物

知识库目录、资料来源表、10 个测试问题、问答验收表。

练习任务

拿一份产品说明和一份客服 FAQ，让 Codex 整理成“新员工可以直接问”的知识库，并测试 5 个问题。

常见错

只把资料复制进去不分类；资料过期也不标记；回答没有来源；把知识库当聊天机器人，不做维护。

9.2 案例二：Codex × 12306 查询工具：学会外部实时查询

教学案例案例二：Codex × 12306 查询工具：学会外部实时查询

12306 查询工具用来训练员工理解“查询型工具”。它不是教大家买票，而是教大家怎么让 Codex 调外部实时数据，并判断结果是否可靠。

学什么

MCP 工具调用、参数填写、实时结果、异常处理。适合车票、库存、订单、物流、价格这类会变化的数据。

准备什么

准备工具说明、可用参数、正常案例、无结果案例、错误案例。先明确工具只能查询，不能替你确认付款或提交敏感操作。

照着做

第一步，写清查询目标；第二步，补齐出发地、目的地、日期等参数；第三步，让 Codex 调工具；第四步，看返回字段；第五步，用异常问题测试它会不会乱答。

验收

正常查询能返回结果；条件不全会追问；无结果会说明没有查到；接口失败会提示失败原因；不会把实时查询说成永久结论。

交付物

工具参数表、正常/异常测试记录、失败提示模板。

练习任务

设计一个“库存查询”练习：让 Codex 查询某个商品是否有货，要求它在缺少城市或门店时先追问。

常见错

问题里缺参数却直接查；接口失败还硬编结果；不区分实时结果和固定知识；没有测试无结果场景。

9.3 案例三：Codex × Chrome：浏览器操作与验收

教学案例案例三：Codex × Chrome：浏览器操作与验收

Chrome 案例训练员工让 Codex 进入真实网页，看页面、点按钮、截图、发现问题。它适合需要登录态、真实页面状态、真实浏览器兼容性的任务。

学什么

浏览器自动操作、截图检查、页面阅读、表单测试、线上页面验收。

准备什么

准备网址、测试账号或已登录浏览器、要检查的页面清单、禁止操作清单。涉及提交、付款、删除、发布时必须先停下来确认。

照着做

第一步，告诉 Codex 要检查哪个网页；第二步，列出要点击的入口；第三步，让它边操作边截图；第四步，把发现的问题按页面记录；第五步，桌面和手机视口各查一次。

验收

能说清打开了哪个页面、点了哪些按钮、截图在哪里、哪里通过、哪里需要改。页面问题要能复现，不只写感觉。

交付物

截图文件、页面问题清单、复现步骤、修复后复查记录。

练习任务

打开一个内部教程页，让 Codex 检查左侧导航、图片显示、手机端排版，并输出截图证据。

常见错

只让 Codex 看文字，不让它真的打开页面；没有截图；没有说明电脑端和手机端分别是否正常；让它乱点危险按钮。

9.4 案例四：Codex × DKFile：本地文件批处理

教学案例案例四：Codex × DKFile：本地文件批处理

DKFile 这类案例训练员工处理本地文件：批量改名、分类、提取信息、核对结果。关键是先预览，再执行，再抽查。

学什么

文件整理、批量处理、处理前后对照、错误回滚。适合图片、表格、合同、素材、导出资料。

准备什么

准备样本文件夹、命名规则、分类规则、备份位置。第一次只拿 5-10 个文件试跑，不要直接处理整个硬盘。

照着做

第一步，让 Codex 列出文件结构；第二步，写清命名和分类规则；第三步，先 dry-run 输出将要改什么；第四步，确认后再执行；第五步，抽查文件能否打开。

验收

处理前后有对照表；文件没有丢；命名规则一致；异常文件被单独列出来；能回到处理前版本。

交付物

处理前清单、dry-run 结果、处理后清单、异常文件列表。

练习任务

找一个图片文件夹，按“日期-主题-序号”重命名。先让 Codex 只输出预览表，再执行。

常见错

一上来处理全部文件；没有备份；没有先预览结果；处理完不知道改了哪些文件。

9.5 案例五：Codex × Draw.io MCP：把流程画出来

教学案例案例五：Codex × Draw.io MCP：把流程画出来

Draw.io MCP 案例训练员工把复杂流程可视化。员工不用先追求图好看，先把开始、步骤、判断、分支、责任人画清楚。

学什么

SOP、业务流程、审批流程、工具调用流程、异常处理流程。

准备什么

准备一段流程文字、涉及角色、开始条件、结束条件、异常情况。越复杂的流程，越要先列节点再画图。

照着做

第一步，让 Codex 从文字里抽节点；第二步，区分动作和判断；第三步，生成流程图；第四步，检查有没有漏分支；第五步，导出给同事确认。

验收

别人不看长文，只看图也能知道先做什么、遇到问题走哪条分支、最后交付什么。

交付物

Draw.io 流程图、节点说明、异常分支清单。

练习任务

把“新员工发布教程网站”的流程画出来，必须包含本地预览、检查、发布、线上复查、回滚。

常见错

图里只有大词，没有动作；判断条件不清楚；流程图和正文说法不一致；没有异常分支。

9.6 案例六：Codex × Figma MCP：读设计稿并落地页面

教学案例案例六：Codex × Figma MCP：读设计稿并落地页面

Figma MCP 案例训练员工把设计稿落地成页面。它只是案例六，不是全部案例。教学重点是：读结构、做页面、截图对比、继续修。

学什么

读设计稿、理解组件、生成页面、检查还原度、移动端适配。

准备什么

准备具体 Figma 文件或 Frame 链接、要落地的页面范围、可接受的差异、字体和颜色要求。只有截图也能参考，但最好有设计稿链接。

照着做

第一步，让 Codex 读取设计稿结构；第二步，确认页面模块和文字；第三步，生成页面；第四步，用浏览器截图对比；第五步，修间距、字体、颜色和响应式。

验收

页面能打开；桌面和手机不乱；主要文字、层级、颜色、间距接近设计稿；无法读取的细节被明确标出。

交付物

页面文件、桌面截图、手机截图、设计稿差异清单。

练习任务

选一个 Figma 页面，让 Codex 做出本地网页，再用截图对比列出 5 个差异并修正。

常见错

只给截图不给链接；只说“照着做”；没有对比截图；没有检查手机端；把 Figma 当成唯一案例。

9.7 案例七：Codex × GitHub Actions：让 CI 帮你守住质量

教学案例案例七：Codex × GitHub Actions：让 CI 帮你守住质量

GitHub Actions 案例训练员工不要只靠肉眼检查。能自动跑的检查，就交给 CI；失败以后，让 Codex 读日志、定位、修复、复跑。

学什么

自动测试、构建检查、失败定位、修复后复跑。适合网站、脚本、文档构建、前端项目。

准备什么

准备 GitHub 仓库、本地能跑通的命令、需要自动检查的项目类型。先保证本地命令成功，再写 Actions。

照着做

第一步，让 Codex 找出本地检查命令；第二步，写最小 Actions 工作流；第三步，push 后看检查结果；第四步，失败就读日志；第五步，修复后复跑。

验收

每次提交都有检查记录；失败原因能定位到命令、文件或依赖；修复后状态变绿；团队知道哪里看结果。

交付物

workflow 文件、一次成功运行记录、一次失败定位说明。

练习任务

给一个网页项目加 GitHub Actions：每次 push 后自动跑构建，并把失败日志整理成中文原因。

常见错

看到红色只截图不看日志；工作流太复杂；本地都跑不通就直接上 CI；失败后不复跑确认。

9.8 案例八：Codex × HyperFrames：把网页和脚本变成教学视频

教学案例案例八：Codex × HyperFrames：把网页和脚本变成教学视频

HyperFrames 案例训练员工把静态教程变成动态说明。它是辅助教学，不是拿来堆特效。

学什么

教学短视频、网页演示、流程动效、培训素材。适合把操作步骤讲给员工看。

准备什么

准备脚本、页面截图、要演示的步骤、时长要求。先写脚本，再做视频，不要一上来做动效。

照着做

第一步，写清视频讲哪一个步骤；第二步，确定画面顺序；第三步，让 Codex 生成视频工程；第四步，本地预览；第五步，检查文字、画面、节奏；第六步导出。

验收

视频能解释一个步骤；画面清楚；文字不遮挡；时长不拖；员工看完知道下一步做什么。

交付物

视频工程、预览截图、导出视频、脚本文案。

练习任务

把“如何检查网页手机端”做成 30 秒教学视频，要求只讲一个动作，不要塞太多内容。

常见错

为了好看加太多动效；没有脚本；画面和讲解不同步；导出后没检查。

9.9 案例九：Codex × LLM Wiki：把资料整理成知识库

教学案例案例九：Codex × LLM Wiki：把资料整理成知识库

LLM Wiki 案例训练员工做资料整理。重点不是把内容变多，而是把散乱资料变成能查、能学、能维护的结构。

学什么

资料分类、目录设计、知识页拆分、来源记录、更新维护。

准备什么

准备散乱资料、旧文档、截图、链接。先明确读者是谁：新人、客服、运营、技术，读者不同目录也不同。

照着做

第一步，收集资料；第二步，删掉无关内容；第三步，按主题建目录；第四步，每页写一句用途；第五步，给关键资料标来源；第六步，补检索入口。

验收

新人知道从哪一页开始；每页主题单一；资料来源清楚；过期内容能被发现；目录能继续扩展。

交付物

知识库目录、页面模板、资料来源表、维护责任表。

练习任务

把 5 篇旧教程整理成一个 LLM Wiki 目录，要求每篇都有“用途、适合谁、更新时间”。

常见错

把所有内容堆在一页；没有目录；没有来源；标题很大但里面没有可执行步骤。

9.10 案例十：Codex × Obsidian：整理个人和团队知识

教学案例案例十：Codex × Obsidian：整理个人和团队知识

Obsidian 案例训练员工做长期知识积累。它不是只做一篇漂亮笔记，而是建立能持续增长的知识结构。

学什么

个人笔记整理、团队知识链接、标签、双链、周复盘。

准备什么

准备一个笔记库、旧笔记、常用主题、标签规则。先定少量一级分类，不要一开始就做复杂体系。

照着做

第一步，定 3-5 个一级分类；第二步，把旧笔记归档；第三步，用标签标状态；第四步，建立常用模板；第五步，每周让 Codex 帮你清理孤立笔记。

验收

能通过目录找到内容；相关笔记互相链接；重复笔记减少；常用模板能复用；每周能维护。

交付物

Obsidian 目录结构、标签规则、笔记模板、清理清单。

练习任务

整理一个“Codex 学习”笔记库，至少包含教程、案例、模板、问题记录四类。

常见错

分类太细；标签随手乱写；只搬运内容不整理；没有定期清理。

9.11 案例十一：Codex × Notion：把笔记变成可分享网站

教学案例案例十一：Codex × Notion：把笔记变成可分享网站

Notion 案例和你这个网站最相关。它训练员工把原始笔记改成清楚、可跳转、可分享的教程页。

学什么

笔记改写、目录跳转、图片排版、网站发布、读者体验优化。

准备什么

准备原 Notion 文案、原图片、参考风格、不能改掉的表达、读者对象。先保留原内容，再优化结构。

照着做

第一步，提取原文和图片；第二步，保留原意；第三步，重排章节；第四步，短句化；第五步，加入导航；第六步，检查桌面和手机；第七步，发布链接。

验收

读者能快速选入口；左侧导航能跳转；图片不压文字；手机端不挤；原文重点没有被改丢。

交付物

可分享网页、导航结构、图片清单、最终验收截图。

练习任务

选一篇乱笔记，改成“新人照着做”的网页，必须有左侧导航、案例图、最终检查表。

常见错

重新编一篇，不基于原文；图片乱放；只优化电脑端；把用户专门写的表达改掉。

9.12 案例十二：Codex × PPT Skill：稳定生成 PPT 文件

教学案例案例十二：Codex × PPT Skill：稳定生成 PPT 文件

PPT Skill 案例训练员工把可重复文件交付做稳。重点不是让模型随便排版，而是让 Skill 和脚本负责确定性生成。

学什么

固定风格 PPT、培训课件、汇报模板、批量生成文件。

准备什么

准备 PPT 模板、品牌颜色、页数、每页标题、图片素材。先确定结构，再生成文件。

照着做

第一步，准备模板；第二步，写清页数和结构；第三步，让 Codex 生成内容大纲；第四步，用脚本生成 PPTX；第五步，打开检查版式；第六步，不合格就改模板或脚本。

验收

能生成真实 PPTX 文件；字体、页边距、标题层级稳定；重新生成不会大幅跑偏；图片不遮挡文字。

交付物

PPTX 文件、生成脚本、模板说明、版式检查记录。

练习任务

把本教程第 1-3 课生成 8 页培训 PPT，要求每页标题不换行、文字不超出。

常见错

只让模型描述 PPT，不生成文件；没有模板；没有打开文件检查；每次风格都不一样。

9.13 案例十三：Codex × Playwright：自动检查网页

教学案例案例十三：Codex × Playwright：自动检查网页

Playwright 案例训练员工做网页验收。它比人工点页面更稳定，尤其适合检查导航、按钮、表单、截图。

学什么

自动化浏览器测试、截图对比、移动端检查、回归测试。

准备什么

准备本地或线上网址、要测试的按钮和链接、桌面/手机尺寸、期望结果。先用人工跑一遍，再写自动化。

照着做

第一步，写要检查的页面和动作；第二步，让 Codex 写 Playwright 脚本；第三步，运行；第四步，保存截图；第五步，失败后定位选择器或页面问题；第六步，修复后复跑。

验收

脚本能打开网页；能完成点击或输入；截图保存成功；失败信息能看懂；桌面和手机都测过。

交付物

Playwright 脚本、桌面截图、手机截图、测试结果。

练习任务

写一个脚本检查本网站：打开首页、点击第 9 课、确认案例十五存在、保存手机端截图。

常见错

没有等待页面加载；选择器随便写；只测一个屏幕宽度；脚本跑完不看截图。

9.14 案例十四：Codex × 飞书 CLI：提取云文档资料和图片

教学案例案例十四：Codex × 飞书 CLI：提取云文档资料和图片

飞书 CLI 案例训练员工做资料迁移。就像这次网站，先把飞书结构、文字、图片提出来，再决定哪些保留、哪些删掉。

学什么

云文档提取、图片迁移、结构映射、内部培训资料重做。

准备什么

准备飞书链接、访问权限、要保留的范围、要删除的范围。先只处理当前页和相关子页，不要把整个知识库都搬进来。

照着做

第一步，抓当前页；第二步，列出模块、图片、子页面；第三步，做映射表；第四步，保留教学相关内容；第五步，删除宣传和无关内容；第六步，生成网站草稿。

验收

有结构清单；图片能打开；无关内容删干净；保留内容能对应到网站章节；没有漏掉侧栏案例。

交付物

结构映射表、图片清单、保留/删除清单、网站草稿。

练习任务

给一篇飞书教程做迁移：输出“原模块 -> 新网站章节 -> 图片用途 -> 是否保留”的表。

常见错

只看第一页；漏掉侧栏案例；图片路径失效；把无关人事、宣传、案例故事也搬进来。

9.15 案例十五：Codex × 云服务：部署分享和回滚

教学案例案例十五：Codex × 云服务：部署分享和回滚

云服务案例训练员工做最后一公里：网站、文档或工具做好以后，怎么让团队能访问，出问题怎么退回。

学什么

部署网站、生成分享链接、检查线上页面、准备回滚。

准备什么

准备本地成品、部署仓库或云服务、访问权限、回滚方式。发布前先确认本地预览没问题。

照着做

第一步，本地预览；第二步，确认资源路径；第三步，部署到 GitHub 或云服务；第四步，打开线上链接检查；第五步，记录发布时间；第六步，保留回滚方式。

验收

领导和员工能打开链接；图片正常；手机端正常；旧版本能找回；发布步骤可复用。

交付物

线上链接、发布时间、版本号、线上检查截图、回滚说明。

练习任务

把一个静态网页部署到 GitHub Pages，发布后检查图片、手机端、导航，并记录如何回滚。

常见错

本地能看，线上图片打不开；没有记录版本；没有回滚；部署后不检查手机端。

10. 团队训练和落地

这份教程最后要服务团队，不是只给一个人看。

10.1 新人 3 天训练

时间学习内容练习交付物
第 1 天第 1-2 课列岗位任务，跑一个小任务完整任务记录
第 2 天第 3-5 课写任务说明卡，做一次验收任务说明卡 + 验收表
第 3 天第 6-9 课选一个工具或案例做练习MCP 判断表 / Skill 草稿 / 案例练习记录

10.2 讲师怎么带

不要先讲概念。

按这个顺序带：

  1. 先让员工说自己的岗位任务。
  2. 再选一个小任务跑通。
  3. 再让 Codex 复述计划。
  4. 再看它执行和验收。
  5. 再指出哪里读着累、哪里没证据。
  6. 最后把流程沉淀成模板或 Skill。

10.3 团队标准

团队内部可以统一 5 个要求：

  • 任何大任务先写任务说明卡。
  • 任何网页类结果必须检查桌面端和手机端。
  • 任何图片必须有用途，不放纯装饰图。
  • 任何外部工具先做风险判断，再接 MCP。
  • 任何可复用流程先手动跑通，再沉淀 Skill。

10.4 最终验收表

维度必须达到不通过就怎么改
教程主线读者知道先学什么、后学什么重排章节顺序
文字短句、步骤、少废话拆段、删空话、加动作
图片图片能降低理解成本换位置或删掉无用图
案例案例只辅助方法补“学什么、照着做、验收、常见错”
工具先判断是否需要，再接入补 MCP 判断表
网页桌面、手机、导航、图片都正常修布局和资源路径
线上员工打开链接看到新版重新部署并查线上资源

10.5 最后记住

Codex 用得好，不是因为某一句话写得神。

是因为你把工作交代清楚，把资料给够，把过程看住，把结果验收，再把好流程沉淀下来。
