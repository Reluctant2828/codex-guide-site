Chapter 01

0. 学习入口

这份教程按“先能用、再用好、最后能教别人”的顺序写。

员工不需要先懂代码，也不需要先懂很多 AI 术语。

只要平时会整理资料、写文档、做表格、检查网页、做 PPT、发内部教程，就可以从这里开始。

0.1 读完以后要会什么

能力读完以后应该会不追求什么
判断任务知道哪些工作适合交给 Codex，哪些还要人工决定不背概念
交代任务能把一句“帮我弄一下”改成目标、材料、限制、验收标准不迷信神奇话术
给资料能说明每个文件、截图、链接的用途不把所有东西乱丢进去
看过程会让 Codex 先读资料、先列计划、再执行不一次性放手不管
用工具知道 App、CLI、IDE、浏览器、Skill、MCP 分别适合什么不为了高级而高级
验收结果会检查截图、文件、命令输出、线上链接和失败记录不只听“完成了”
团队复用能把好流程沉淀成检查表、AGENTS.md、Skill 或培训练习不让每个人都各用各的

0.2 为什么不是提示词合集

只复制一句话，员工很快就会忘。

真正有用的是一套工作方法：

  • 先判断这个任务适不适合交给 Codex。
  • 开始前准备目标、资料、限制和验收标准。
  • 执行前让 Codex 复述理解和计划。
  • 执行中让它留下证据，不要只给结论。
  • 结果出来后按清单验收。
  • 不满意就继续狠狠 push：指出哪里不对，让它基于证据继续改。
温馨提示

如果员工只学到一句话，他只能完成一次任务。如果员工学会“任务卡 + 资料清单 + 过程检查 + 验收标准”，换成文档、网页、PPT、表格、知识库都能继续用。

0.3 这份教程怎么读

你现在的状态先看哪里马上练什么做完交什么
第一次用 Codex第 1 到 2 课跑通一个小任务完整任务记录 + 截图
经常说不清需求第 3 课把模糊需求改成任务卡任务卡一张
资料很多很乱第 4 课做资料映射表资料清单 + 保留删除清单
不知道用什么工具第 5 课做工具选择判断工具选择表
结果经常不稳第 6 到 7 课让 Codex 自查和复测验收记录
要培训团队第 8 到 10 课选 2 个案例演示讲师流程 + 员工练习记录

0.4 飞书资料怎么用到这里

飞书文档好的地方是结构完整：有入口、有侧栏、有案例、有图片、有步骤。

但我们不照搬所有内容。

我们只保留对 Codex 学习有用的部分。

飞书内容类型怎么处理
首页入口和导航节奏保留，改成 Codex 学习入口
AI / Agent / MCP 长篇理论压缩成普通员工能懂的一句话和使用判断
Codex 实战案例全部放进第 9 课，改成训练任务
图片和截图只放能帮助理解的位置，不做纯装饰
宣传、无关案例、其他工具长文不进入正文
Chapter 02

1. 第一课：先知道 Codex 到底是什么

Codex 可以先理解成“会动手的 AI 工作助手”。

普通聊天工具主要回答你。

Codex 更适合接任务：读文件、看图片、改内容、跑命令、打开网页、检查结果、把成果保存成真实文件。

1.1 Codex 和普通聊天有什么不同

对比普通聊天Codex
输入你问一句，它答一句你给目标、资料、限制和验收标准
过程主要靠文字回答可以读文件、改文件、跑命令、看网页
结果一段答案真实文件、网页、截图、报告、脚本、检查结果
适合任务解释、翻译、头脑风暴整理资料、做网站、写文档、批处理、检查问题
你要做什么判断答案有没有道理管理任务过程并验收交付物

1.2 一句话理解 AI、LLM、Agent、Codex

词员工只需要先这样理解
AI一类能帮你处理信息和任务的技术
LLM很会理解和生成文字的大模型
Agent能围绕目标观察、行动、检查、再行动的工作方式
Codex把 Agent 能力放到文件、网页、代码、文档、工具里的工作助手

1.3 Codex 最适合的工作

工作类型适不适合为什么
整理资料很适合它能读多份材料，提炼结构，输出清单
改网页或文档很适合它能直接改文件并让你验收
检查页面很适合它能打开浏览器、截图、列问题
批量处理文件很适合它能先试跑，再全量处理
生成 PPT / 报告适合，但要验收内容和结构可以交给 Codex，版式必须检查
决定公司战略不适合直接交给它它可以辅助分析，最终判断必须由人做
涉及隐私或高风险操作谨慎先脱敏、先确认权限、先小范围试跑

1.4 Codex 的几个常见入口

入口适合什么员工怎么选
Codex App整理资料、写教程、看网页、做多步任务不知道从哪开始时先用它
CLI本地项目、命令行、脚本、批处理任务和文件夹强相关时用
IDE 扩展边看代码边改、修 bug、做小功能正在编辑项目时用
Browser / Chrome真实网页、登录态页面、截图验收结果必须打开网页看时用
Computer Use桌面软件和系统界面必须操作本机软件时用
Skill固定流程反复做流程跑通后再沉淀
MCP连接外部系统或工具资料在 Figma、GitHub、文档库、浏览器里时考虑
Chapter 03

2. 第二课：第一次完整使用 Codex

第一次不要上来就做大项目。

先选一个 15 到 30 分钟能完成的小任务。

目标是学会流程，不是一次做得多漂亮。

2.1 选择一个最小任务

好任务长这样：

  • 范围小：一篇文档、一个网页、一个文件夹、一个表格。
  • 资料齐：有原文、图片、参考页面或现成文件。
  • 结果清楚：做完应该有文件、截图、清单或链接。
  • 能验收：你知道什么算通过，什么算不通过。

不建议第一次就做：

  • “帮我做一个完整系统”。
  • “你自己看着发挥”。
  • “把所有资料都整理一下”。
  • “做得高级一点”。

2.2 第一次任务的标准流程

步骤你要做什么Codex 要给什么证据
1. 说目标告诉它最终要交付什么复述目标
2. 给材料说明文件、图片、链接分别有什么用资料清单
3. 说限制告诉它不能改什么、不能删什么、风格是什么风险点和假设
4. 先要计划让它先列步骤，不要直接开干执行计划
5. 执行让它边做边说明关键改动改动清单
6. 验收让它打开、截图、跑检查或列对照截图、命令输出、检查表
7. 追改指出不满意的地方继续 push修正说明和复测结果

2.3 任务卡怎么写

任务卡不是“提示词模板”，而是工作交接单。

员工每次开任务，至少写清这 5 件事：

项目要写清楚什么例子
目标最后要交付什么把这篇笔记改成给新人看的网页教程
材料哪些文件、图片、链接有用原 Notion 文案、参考飞书截图、chikiwa 贴纸
限制不能改什么、不要什么保留原意，不要加入无关工具，不要删掉用户专门写的表达
过程先做什么再做什么先读原文，再列结构，再改正文，再验收网页
验收什么叫完成桌面和手机正常，图片不压文字，导航能跳转
温馨提示

任务卡写得越清楚，Codex 越像秘书：知道要先整理资料、再安排步骤、最后把结果交给你验收。任务卡写得模糊，它就只能猜。

2.4 第一次任务示范

场景正确做法
你要改一篇乱笔记先让 Codex 提取原文结构和图片清单，不要让它直接重写
你要做一个网页先让它说明布局，再生成页面，再用浏览器截图检查
你要做 PPT先定页数和结构，再生成 PPTX 文件，最后打开检查版式
你要整理文件先扫描文件并试跑 3 个样本，再全量处理

2.5 第一次任务的结束标准

一次任务真正完成，必须有证据。

  • 有最终文件、网页、表格、PPT、截图或链接。
  • 有改动说明，不是只说完成。
  • 有验收记录，例如截图、命令输出、检查清单。
  • 如果有失败，有失败原因和处理方式。
  • 你能把成果发给别人，不需要再解释半天。
Chapter 04

3. 第三课：把一句愿望变成可执行任务

很多人用不好 Codex，不是因为不会写漂亮话。

是因为任务本身太模糊。

3.1 模糊任务怎么改

模糊说法为什么不行改成可执行任务
帮我优化一下不知道优化什么把这篇教程改成新人能照着做的版本，保留原意，减少长段落
做得好看一点没有判断标准参考图一的侧栏结构，电脑端不留大空白，手机端不横向滚动
内容多一点可能变成堆字补齐“用途、步骤、验收、常见错”，删掉无关案例
你自己发挥容易跑偏先提出结构方案，等我确认后再改正文和网页
全部整理一下范围太大先只处理当前页和侧栏 Codex 案例，输出保留/删除映射表

3.2 一个好任务包含四层

层意思没有它会怎样
目标你到底要什么结果Codex 会自己猜方向
上下文相关文件、图片、链接、背景结果容易凭空编
限制不能改什么、不要什么、风格边界会把你专门写的表达改掉
完成标准什么证据证明完成最后只能听它说“好了”

3.3 追问和狠狠 push

不满意时，不要只说“不行”。

要指出问题属于哪一类：

问题怎么继续 push
读起来累把长段拆短，每段只讲一个动作，表格只保留决策信息
像说明书增加“先做什么、再做什么、做完看什么”的步骤
像复制内容删掉可复制话术，改成任务流程和验收标准
案例喧宾夺主把案例放到练习区，正文先讲使用方法
图片没用说明图片在这一节帮助读者理解什么，不需要就删
网页乱用桌面和手机截图逐项检查布局

3.4 大任务怎么拆

大任务不要一次扔给 Codex。

按这四段拆：

阶段目标交付物
盘点先弄清楚有什么资料资料清单、图片清单、章节清单
设计先定新结构目录、模块映射、保留删除表
制作再改正文和网页Markdown、HTML、图片排版
验收最后检查读者体验桌面截图、手机截图、问题清单、线上链接
Chapter 05

4. 第四课：让 Codex 读懂你的资料

资料不是越多越好。

资料要有顺序、有用途、有边界。

4.1 资料先分类

资料类型你要告诉 Codex 什么
原文这是必须保留原意的内容
参考这是风格或结构方向，不是照抄内容
图片说明哪个步骤或哪个案例
禁止范围这些内容不要进入正文
读者背景读者是谁，懂什么，不懂什么
验收标准读者看完应该能做什么

4.2 资料映射表

处理飞书、Notion、长文档时，先做映射表。

不要直接重写。

原资料模块新网站放到哪里图片怎么用是否保留
入口页0. 学习入口保留入口节奏保留
Agent 基础长文1. Codex 是什么只放一张循环图压缩
Prompt 工程长文3. 任务交代方法不堆模板只保留方法
MCP 长文5. 工具选择放一张架构图压缩
Codex 案例侧栏9. 完整案例训练库保留案例图保留

4.3 图片不是装饰

图片必须回答一个问题：它帮读者理解什么？

图片用途适合放哪里
说明目录结构学习入口或案例总览
说明工作循环Codex 基础认知
说明工具连接MCP / Skill / 工具选择
说明案例结果对应案例内部
只是好看不要放

4.4 从飞书提取内容时怎么筛

筛选原则很简单：

  • 能帮助员工学会 Codex 的，保留。
  • 只是大篇幅介绍概念的，压缩。
  • 只是展示成功故事的，不进正文。
  • 其他工具和无关方向，不进正文。
  • 图片能降低理解成本，保留；只是装饰，删掉。
Chapter 06

5. 第五课：插件、Skill、MCP 和自动化怎么选

很多人看到插件、Skill、MCP、自动化就开始乱。

先记住一句话：

温馨提示

先用最简单的方式跑通任务。任务跑通以后，如果反复要做，再考虑 Skill、MCP 或自动化。

5.1 工具选择总表

你遇到的情况先选什么什么时候升级
只是改一篇文章或教程普通 Codex 任务反复同类文章再做 Skill
要看网页效果Browser / Chrome反复验收网页再写 Playwright
要处理本地文件CLI 或 App批量固定流程再做脚本或 Skill
资料在 Figma / GitHub / 文档库先给链接或截图长期需要读取再接 MCP
每周都要同样格式输出先人工跑通一次稳定后做 Skill 或自动化
要团队统一规则AGENTS.md / 检查表规则稳定后再包装成 Skill

5.2 插件怎么理解

插件可以理解成给 Codex 装工具箱。

它解决的是“Codex 现在多了哪类能力”。

用插件前先问：

  • 它能处理我现在这个任务吗？
  • 它需要什么权限或文件？
  • 它输出什么结果？
  • 结果怎么验收？

5.3 Skill 怎么理解

Skill 适合“反复做，而且有固定标准”的事情。

它不是一句更长的话，而是一套可复用工作流程。

值得做成 Skill暂时不值得
每周都要做只做一次
步骤固定每次都完全不一样
有明确验收标准结果靠感觉
可以配脚本或模板稳定输出只是想把话写长一点
别人也会用只有你自己偶尔用

5.4 MCP 怎么理解

MCP 可以理解成连接线。

当资料或工具在 Codex 默认够不着的地方，就考虑 MCP。

系统为什么可能需要 MCP例子
Figma设计稿在 Figma 里，截图不够准确读取 frame、组件、颜色、间距
GitHub代码、Issue、PR、Actions 都在仓库里读失败日志、创建 PR、检查 CI
内部知识库资料在公司系统里检索制度、文档、历史记录
浏览器或网页工具需要真实页面状态打开后台、截图、检查按钮

5.5 AGENTS.md 什么时候用

AGENTS.md 适合放团队项目里的长期规则。

比如：

  • 项目怎么启动。
  • 改完要跑哪些检查。
  • 文件命名规则。
  • 哪些目录不能乱动。
  • 什么叫完成。

如果每次都要重复告诉 Codex 同样规则，就应该沉淀进 AGENTS.md 或团队检查表。

5.6 自动化什么时候用

自动化只适合稳定流程。

流程不稳定时先别自动化。

适合自动化不适合自动化
每天检查网页是否能打开需求还没定清楚
每周生成固定格式汇总每次格式都变
定时提醒某件事需要复杂人工判断
重复跑同一套验收脚本还不知道怎么验收

6. 第六课：把复杂任务拆成工作流

Codex 不是一次性按钮。

复杂任务要让它按阶段推进。

6.1 标准工作流

阶段你要让 Codex 做什么你要检查什么
理解复述目标、资料、限制有没有误解
盘点列出文件、图片、页面、风险有没有漏资料
计划拆成步骤和顺序顺序是否合理
执行按步骤改文件或生成结果有没有证据
自查自己找问题有没有只说好话
验收跑命令、截图、打开文件、检查链接是否达到完成标准
复盘记录下次可复用的方法能不能沉淀

6.2 什么时候必须先要计划

以下任务不要让 Codex 直接开干：

  • 会改多个文件。
  • 会删除、迁移、批量处理内容。
  • 会发布到线上。
  • 会接外部工具或权限。
  • 会影响别人使用。
  • 用户明确说“不要漏”。

6.3 让 Codex 自己 review

每次交付前，让 Codex 从读者、维护者、验收者三个角度自查。

角色要检查什么
读者能不能看懂，是否像教程，不像说明书
维护者结构是否清楚，以后能不能继续改
验收者有没有截图、命令输出、线上链接、失败记录

6.4 失败了怎么办

失败不是结束。

失败后要让 Codex 做三件事：

  • 说明失败发生在哪一步。
  • 列出它已经确认过的事实。
  • 提出下一步最小修复动作。

不要接受“可能是网络问题”这种空话。

要让它给证据。

7. 第七课：验收不是最后随便看一眼

Codex 做完后，最重要的是验收。

没有验收，就不能发给员工或领导。

7.1 不同成果怎么验收

成果验收方式
网页桌面截图、手机截图、导航跳转、图片加载、横向滚动检查
文档目录是否清楚、原意是否保留、是否有长段难读内容
PPT打开文件、检查每页标题、图片遮挡、字体和页边距
表格行列数量、公式、异常值、导出文件
文件批处理处理前后对照表、样本检查、异常文件清单
线上发布真实链接、缓存参数、图片路径、回滚方式

7.2 网页验收清单

  • 电脑端右侧不要空一大块。
  • 手机端不能横向滚动。
  • 标题不要出现第二行只有一两个字。
  • 图片不能压住文字。
  • 左侧导航能跳转。
  • 案例卡片不应该比正文主线更抢戏。
  • 线上链接必须是最新版。

7.3 让员工自己也能验收

验收要傻瓜式。

员工只要照着问：

问题通过标准
我知道先看哪一节吗？首页有明确入口
我知道这一节学什么吗？每节开头说用途
我知道下一步怎么做吗？有步骤，不只是概念
我知道做完交什么吗？有交付物
我知道哪里算错吗？有常见错误

8. 第八课：把好用的方法沉淀下来

团队使用 Codex，不能只靠每个人临场发挥。

要把好方法留下来。

8.1 三种沉淀方式

方式适合放什么例子
检查表人人都要遵守的简单规则网页发布前检查桌面和手机
AGENTS.md项目里的长期规则启动命令、测试命令、文件规范
Skill跨项目重复使用的完整流程飞书资料改网站、PPT 生成、会议纪要整理

MCP外部系统连接能力Figma、GitHub、内部文档库

8.2 一个流程什么时候值得沉淀

满足三条就可以沉淀：

  • 已经人工跑通过一次。
  • 下次还会再做。
  • 有明确验收标准。

不要一上来就做 Skill。

先跑通，再沉淀。

标准具体要求
开工标准任何大任务先写任务卡
资料标准原文、参考、图片、禁止范围分开
过程标准先复述、再计划、再执行
验收标准必须有截图、文件、链接或命令输出
发布标准线上链接、发布时间、回滚方式都要记录
复盘标准高频流程写成检查表或 Skill

9.1 案例一：Codex × PPT Skill：一句话生成演示文稿

案例一训练案例
案例一：Codex × PPT Skill：一句话生成演示文稿

训练员工安装并调用一个社区 Skill，用真实主题生成演示稿。重点不是“让 AI 写 PPT”，而是学会安装、补资料、生成、预览和检查。

案例定位Skill / PPT 生成
做完要交Skill 安装结果、调用说明、演示稿文件、预览截图、修改记录。
学什么

社区 Skill 安装、依赖识别、输入补齐、演示稿生成、预览检查。

准备什么

准备 Skill 仓库链接、演示主题、受众、页数、风格、必须出现的内容和输出格式。

照着做

交付物

Skill 安装结果、调用说明、演示稿文件、预览截图、修改记录。

练习任务

用“AI 编程工具入门”做一份 8 页分享稿，要求先补齐受众、页数和风格，再生成文件。

常见错

只让 Codex 描述 PPT，不生成文件；没有看 Skill README；没打开预览；结果不好时不知道是输入不足还是 Skill 能力限制。

案例二：Codex × Draw.io MCP：AI 自动绘制架构图

交付物

练习任务

常见错

所有内容堆一页；没有原始资料区；没有来源；目录一开始就分得太细。

Chapter 11

10. 团队训练和落地

这份教程最后要服务团队。

不是让一个人看完觉得懂了，而是让员工能照着做，讲师能带，领导能验收。

10.1 新人 3 天训练

时间学习内容练习交付物
第 1 天第 1 到 2 课跑通一个小任务完整任务记录
第 2 天第 3 到 5 课写任务卡，做工具选择任务卡 + 工具选择表
第 3 天第 6 到 9 课任选一个案例练习验收截图 + 案例练习记录

10.2 讲师怎么带

讲师不要先讲概念。

按这个顺序带：

  1. 先让员工说自己的岗位任务。
  2. 选一个小任务跑通。
  3. 让 Codex 复述目标和计划。
  4. 看它执行和验收。
  5. 指出哪里读着累、哪里没有证据。
  6. 让员工继续狠狠 push 到能交付。
  7. 最后把流程沉淀成检查表或 Skill。

10.3 最终验收表

维度必须达到不通过怎么改
教程主线读者知道先学什么、后学什么重排章节顺序
文字短句、步骤、少废话拆段、删空话、加动作
图片图片能降低理解成本换位置或删掉无用图
案例案例辅助方法，不抢主线补“学什么、照着做、验收、常见错”
工具先判断是否需要，再接入补工具选择表
网页桌面、手机、导航、图片都正常修布局和资源路径
线上员工打开链接看到新版重新部署并查线上内容

10.4 后续怎么维护

什么时候更新怎么更新
Codex 出了新插件或新功能先判断它解决什么工作问题，再放进第 5 课或第 8 课
团队跑出新案例不要只放截图，要补用途、步骤、验收和常见错
员工反馈读不懂先改对应章节的步骤和例子，不要先加更多概念
网页排版变乱先本地截图检查，再改 CSS，最后部署线上验证
某个案例过时标记待更新，或换成更贴近团队工作的案例

10.5 最后记住

Codex 用得好，不是因为某一句话写得神。

是因为你把工作交代清楚，把资料给够，把过程看住，把结果验收，再把好流程沉淀下来。
